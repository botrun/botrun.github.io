絲滑動畫技能 v2

> Nano Banana 2 Session 延續圖片 + ffmpeg 最佳化 GIF

核心原理

`
原始 4 張圖: [A] --------- [B] --------- [C] --------- [D]
↓ ↓ ↓
Nano Banana 2: 生成 3 張過渡幀 生成 3 張過渡幀 生成 3 張過渡幀
↓ ↓ ↓
最終序列: [A] a1 a2 a3 [B] b1 b2 b3 [C] c1 c2 c3 [D]
(共 16 張 → 絲滑 GIF)
`

關鍵技術：Gemini Multi-Turn Session
- 模型 ID：gemini-3.1-flash-image-preview（Nano Banana 2）
- 每次 API 呼叫傳入前後兩張 keyframe（via inlineData）
- 指定過渡進度百分比（33%、67% 等）
- 模型根據兩張圖的差異生成中間幀，保持角色/風格一致性
- REST API 需手動管理 thought signatures（SDK chat 自動處理）

何時使用

- 用戶說「絲滑動畫」「silky animation」「做動畫」「圖片變動畫」
- 有 2-8 張連續插圖/照片想做成流暢動畫
- 目標平台：Google Chat（GIF 自動播放，上限 200MB）
- 也適用：LINE、Slack、Discord、Email

參數說明

| 參數 | 說明 | 預設值 |
|------|------|--------|
| --images | 逗號分隔的圖片路徑（必填，至少 2 張） | 必填 |
| --style | 藝術風格描述（英文為主） | 無 |
| --frames | 每兩張之間插幾張過渡幀 | 3 |
| --subtitles | 逗號分隔的字幕（對應每張原始圖） | 無 |
| --duration | 每張原始圖顯示秒數 | 2.5 |
| --fps | 幀率 | 12 |
| --width | 輸出寬度（px） | 480 |
| --fade | 轉場秒數 | 0.6 |
| --output | 輸出路徑 | ./story.gif |

Gemini Multi-Turn Session 技術細節

REST API 結構（手動管理 thought signatures）

`json
{
"contents": [
{
"role": "user",
"parts": [
{ "text": "Here is the first keyframe image (Image A):" },
{ "inlineData": { "mimeType": "image/jpeg", "data": "" } }
]
},
{
"role": "model",
"parts": [
{ "text": "I can see Image A. Please provide Image B and your instructions." }
]
},
{
"role": "user",
"parts": [
{ "text": "Here is the second keyframe image (Image B):" },
{ "inlineData": { "mimeType": "image/jpeg", "data": "" } },
{ "text": "Generate transition frame at 50% progress between A and B..." }
]
}
],
"generationConfig": {
"responseModalities": ["IMAGE"],
"temperature": 0.4
}
}
`

Thought Signatures 注意事項

- gemini-3.1-flash-image-preview 的 model response 中，每個 part 可能帶 thoughtSignature
- 若要做 3+ 輪的真正 session 延續，必須將 thoughtSignature 原封不動傳回
- 目前腳本使用「每段獨立 2 輪」策略（避免 signature 管理複雜度）
- 未來可升級為 SDK chat 模式（自動管理 signatures）

與香蕉繪圖技能的關係

- 香蕉繪圖技能：生成單張圖片
- 絲滑動畫技能：接收多張圖片 → 插值過渡 → 組裝動畫

地雷紀錄

地雷 1：ffmpeg zoompan 幀數爆炸 — 2026-03-16

- 問題：-loop 1 輸入 + zoompan 導致幀數 ×75 倍，GIF 300MB+
- 正確：用 -framerate $FPS 控制輸入幀率，或改用 concat demuxer
- 驗證：輸出前檢查 duration 和 file size

地雷 2：ffmpeg drawtext CJK 編碼 — 2026-03-16

- 問題：filter_complex 內直接寫中文字幕會報 "text must be provided" 錯誤
- 正確：用 textfile= 參數指向外部文字檔
- 驗證：確認 .txt 檔案以 UTF-8 編碼寫入

地雷 3：Gemini 2.0 Flash 已過時 — 2026-03-16

- 問題：不要用 gemini-2.0-flash-exp-image-generation，已是舊版
- 正確：使用 gemini-3.1-flash-image-preview（Nano Banana 2）
- 驗證：檢查 MODEL 變數

地雷 4：thoughtSignature 是 camelCase — 2026-03-16

- 問題：研究文件寫 thought_signature（snake_case），但實際 API 回傳是 thoughtSignature（camelCase）
- 正確：REST API response/request 都用 camelCase thoughtSignature
- 驗證：debug 時用 Object.keys(part) 檢查實際欄位名稱
- 影響：漏傳 thoughtSignature 會導致 400 INVALID_ARGUMENT 錯誤

地雷 5：ffmpeg concat 需要絕對路徑 — 2026-03-16

- 問題：concat demuxer 的 file 列表用相對路徑，ffmpeg 找不到檔案
- 正確：所有路徑用 path.resolve() 轉為絕對路徑
- 影響：MP4/GIF 產出為 0 或 ENOENT

地雷 6：序列算圖太慢 — 2026-03-16

- 問題：3 段 segment 序列執行，每段 ~15 秒，總共 45 秒
- 正確：用 Promise.all 平行發射所有 segment，速度快 N 倍
- 關鍵：每段 segment 獨立（不同 keyframe pair），可安全平行

地雷 7：不要刪除過渡幀 — 2026-03-16

- 問題：腳本自動 rmSync(tmpDir) 刪除所有 AI 過渡幀，用戶無法檢查品質
- 正確：保留 _silky_tmp/ 目錄，並產出 frames/ 流水號目錄供檢閱
- 關鍵：用戶需要逐張檢查過渡幀品質，決定是否需要重新生成

地雷 8：過渡幀需要流水號重命名 — 2026-03-16

- 問題：過渡幀命名 seg0_frame0.png 不直覺，用戶難以判斷播放順序
- 正確：產出 frames/ 目錄，按播放順序 01-keyframe-A.jpg、02-transition-A-B-33.png 等
- 範例：--frames 2 + 4 張原圖 → 10 張流水號檔案

地雷 9：純 concat 硬切不夠絲滑 — 2026-03-16

- 問題：ffmpeg concat demuxer 只是硬切（幀跳幀），沒有 crossfade
- 正確：改用 HTML + CSS crossfade + Playwright 錄影，或 ffmpeg xfade
- 最佳方案：HTML 方式可加落葉等裝飾動畫，效果遠超純 ffmpeg
- HTML 錄影流程：HTML（CSS 動畫 + 落葉）→ Playwright recordVideo → WebM → ffmpeg → MP4/GIF

地雷 10：GIF 幀數需要裁切 — 2026-03-16

- 問題：Playwright 錄影尾段有多餘空白幀，GIF 太長
- 正確：用 ffmpeg -t 裁切到指定秒數/幀數
- 範例：ffmpeg -y -i input.mp4 -t 13.17 ...（158 幀 @ 12fps）

進階：HTML 錄影模式

除了純 ffmpeg 組裝，還支援 HTML + Playwright 錄影模式：

`
10 張幀圖 → silky-player.html（CSS crossfade + 落葉動畫 + 字幕）
↓
Playwright recordVideo（headless Chrome）
↓
WebM → ffmpeg → MP4 + GIF
`

優點：CSS 動畫比 ffmpeg 更絲滑、可加落葉/粒子/暈影等裝飾
HTML 位置：與 frames/ 同目錄的 silky-player.html
錄影腳本：record-video.mjs（需要 Playwright 環境）
