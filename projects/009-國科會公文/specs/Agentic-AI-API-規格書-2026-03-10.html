A. API 核心（10 項）

A-001 文字輸入 API

Given 廠商前端傳來一段使用者文字訊息
When  POST /api/v1/chat
Then  回傳 200，開始串流輸出 AI 回應
  • 輸入：{ "user_id": string, "message": string }
  • 輸出：SSE 串流文字

A-002 串流輸出

Given API 收到合法請求
When  AI 開始生成回應
Then  以 Server-Sent Events (SSE) 逐 token 串流回傳
      且每個 chunk 包含 { "delta": string, "done": boolean }

A-003 上傳舊公文

Given 使用者或管理者要上傳舊公文 PDF
When  POST /api/v1/documents/upload (multipart/form-data)
Then  PDF 存入對應資料夾（共用或個人）
      且回傳 { "document_id": string, "tags": string[] }

A-004 下載生成公文

Given AI 已生成完整公文（.md 格式）
When  GET /api/v1/documents/{id}/download?format=docx|pdf
Then  回傳對應格式的檔案

A-005 多輪對話

Given 使用者正在進行第 N 輪公文修改
When  POST /api/v1/chat 帶入 session_id
Then  AI 記得前 N-1 輪的對話脈絡
      且每輪產出一份 .md 中間檔

A-006 Session 管理

Given 使用者開始一個新的公文任務
When  POST /api/v1/sessions
Then  建立新 session，回傳 { "session_id": string }
Given 使用者結束任務
When  DELETE /api/v1/sessions/{id}
Then  session 標記結束，對話歷史保留至永恆知己

A-007 認證授權

Given 廠商系統帶入 API Key + 使用者身份
When  任何 API 請求
Then  驗證 API Key 有效
      且確認 user_id 有權限存取該資源

A-008 錯誤處理

Given API 收到無效請求
When  缺少必填欄位 / 格式錯誤 / 權限不足
Then  回傳標準錯誤格式 { "error": { "code": string, "message": string } }
      且 HTTP status code 正確（400/401/403/404/500）

A-009 HTTPS Proxy 相容

Given 部署環境設有 HTTPS_PROXY / HTTP_PROXY
When  API 對外發出請求（如呼叫 LLM）
Then  自動使用 Proxy 環境變數

A-010 速率限制

Given 單一使用者短時間內大量請求
When  超過速率限制
Then  回傳 429 Too Many Requests
      且 header 包含 Retry-After

B. 後端 — 永恆知己記憶架構（12 項）

B-001 Folder Tree 結構

Given 一個新使用者首次使用系統
When  系統初始化該使用者空間
Then  建立以下 folder tree：
      ├── shared/          ← 共用舊公文模板（管理者維護）
      ├── personal/        ← 使用者個人記憶（永恆知己）
      ├── sessions/        ← 對話歷史
      └── CLAUDE.md        ← 系統提示（不可修改）

B-002 共用資料夾（管理者維護）

Given 管理者要增添新的公版舊公文模板
When  上傳 PDF 至 shared/ 資料夾
Then  所有使用者下次對話時自動可參考新模板
      且支援從 100 → 200 → 300+ 份持續擴充

B-003 個人資料夾（永恆知己）

Given 使用者生成了一份新公文
When  公文確認完成
Then  自動存入 personal/ 資料夾
      且帶有時間戳與標籤

B-004 時間區間過濾 — 對話歷史

Given AI 要參考過去對話
When  檢索對話歷史
Then  只參考最近 1 週內的對話
      且超過 1 週的對話不納入 context

B-005 時間區間過濾 — 使用者公文

Given AI 要參考使用者過去生成的公文
When  檢索 personal/ 資料夾
Then  只參考最近 8 週內的公文
      且超過 8 週的公文不納入 context

B-006 智慧標籤分類

Given 使用者上傳或生成一份公文
When  公文存入系統
Then  AI 自動分類並貼上標籤
      （如：核定經費、人事異動、採購案、開會通知）

B-007 標籤篩選檢索

Given 使用者說「請你參考核定經費相關的舊公文」
When  AI 檢索參考資料
Then  只帶入標籤含「核定經費」的公文
      且結合時間區間過濾

B-008 CLAUDE.md 不可修改

Given 系統部署完成
When  任何人嘗試修改 CLAUDE.md
Then  拒絕修改，回傳 403
      （確保系統提示穩定不被汙染）

B-009 記憶增刪修權限

Given 使用者或廠商管理者要管理個人記憶
When  CRUD personal/ 資料夾內的檔案
Then  使用者可增刪修自己的檔案
      且廠商管理者有權限管理所有使用者的 personal/

B-010 公文中間檔版本管理

Given 使用者進行多輪迭代（第 1 輪 → 第 N 輪）
When  每輪對話完成
Then  產出一份 .md 中間檔，檔名含輪次與時間戳
      例：session-abc_round-3_2026-03-10T14-30.md

B-011 最終輸出轉檔

Given 使用者確認公文最終版本
When  請求最終輸出
Then  同時生成 .docx 和 .pdf 兩種格式

B-012 技能集架構（Skill）

Given 系統要擴展新的應用場景
When  新增一個 skill 資料夾（如：會議紀錄 skill）
Then  API 自動載入新技能
      且不需要修改核心程式碼
      且共用永恆知己記憶架構

C. 前端整合介面（6 項）

C-001 文字輸入框

Given 廠商前端頁面
When  使用者輸入文字
Then  透過 WebSocket 或 HTTP 送至 API

C-002 串流顯示 AI 回應

Given API 回傳 SSE 串流
When  前端接收 delta chunks
Then  即時顯示 AI 生成的文字（打字機效果）

C-003 PDF 上傳介面

Given 使用者要上傳舊公文
When  拖拉或選擇 PDF 檔案
Then  呼叫 A-003 上傳 API
      且顯示上傳進度與成功/失敗狀態

C-004 公文下載

Given AI 生成完整公文
When  使用者點擊下載
Then  提供 Word (.docx) 和 PDF 兩種格式選擇

C-005 使用者回傳修改版

Given 使用者下載 Word 修改後要回傳
When  上傳修改後的 .docx
Then  系統讀取修改內容，AI 可基於修改版繼續迭代

C-006 新專案 / 繼續專案

Given 使用者進入公文系統
When  選擇「新增公文」或「繼續編輯」
Then  新增 → 建立新 session（A-006）
      繼續 → 載入既有 session 的對話脈絡

D. 公文排行榜 — 多維度打分（8 項）

D-001 五維度評分定義

Given 一份 AI 生成的公文 + 一份標準答案公文
When  執行評分
Then  產出以下五個維度的分數（0-100）：

| 編號 | 維度 | 說明 | 權重 |
|------|------|------|------|
| D1 | 格式正確性 | 發文字號、主旨/說明/辦法結構、附件格式 | 25% |
| D2 | 內容完整性 | 所有必填欄位（人名、金額、期程）正確帶入 | 25% |
| D3 | 語氣適切性 | 公文用語正式得體（函請、惠請、奉核） | 20% |
| D4 | 邏輯連貫性 | 主旨→說明→辦法邏輯鏈完整不矛盾 | 15% |
| D5 | 可用性 | 不需修改即可直接使用的程度 | 15% |

D-002 加權總分計算

Given 五個維度各有分數
When  計算總分
Then  總分 = Σ(維度分數 × 權重)
      且四捨五入至小數一位

D-003 扣分原因標註

Given 任一維度低於 90 分
When  輸出評分結果
Then  附上中文扣分原因列表
      例：["缺少發文字號", "附件格式不符"]

D-004 判定等級

Given 計算完加權總分
When  輸出最終判定
Then  總分 ≥ 80 → PASS（可直接使用）
      60 ≤ 總分 < 80 → NEEDS_REVISION（需人工修改）
      總分 < 60 → FAIL（需重新生成）

D-005 模型比較

Given 同一份公文任務
When  分別用 Agentic AI 和 TAIDE（無 Agentic）生成
Then  兩個模型的五維度分數並排呈現
      且標示哪個模型在哪個維度勝出

D-006 批量評分

Given 100 份國科會公文測試集
When  執行批量評分
Then  每份公文產出完整評分結果
      且彙整為排行榜表格

D-007 排行榜表格輸出

Given 批量評分完成
When  產出排行榜
Then  格式包含：
      | 排名 | 模型名稱 | 總分 | 格式 | 內容 | 語氣 | 邏輯 | 可用性 |
      且按總分由高到低排序

D-008 排行榜視覺化

Given 排行榜數據
When  輸出給工研院 AIEC 展示
Then  產出可嵌入簡報的圖表（PNG/SVG）
      且雷達圖呈現各維度差異

E. 高品味題庫（5 項）

E-001 六欄位定義

Given 要建立一個高品味題庫
When  填寫題庫
Then  必須包含以下六欄位：
      1. 動機（為什麼要寫這份公文）
      2. 效益（這份公文達成什麼效益）
      3. 提問（使用者的完整問題，含所有欄位值）
      4. 答案（期望的公文產出）
      5. 原始資料（參考 PDF 或其他文件）
      6. 鑑別原則（審查專家的審查標準）

E-002 提問欄位必須包含完整參數

Given 公文類型為「核定補助」
When  填寫提問欄位
Then  必須包含：教授名稱、金額、計畫名稱、
      研究機構、期程、其他必填欄位
      （即使模擬也必須像真實資料）

E-003 多 Sheet 格式

Given 一份高品味題庫試算表
When  包含多個任務
Then  每個 Sheet 是一個完整任務
      且 Sheet 名稱為任務摘要

E-004 多輪對話紀錄

Given 一個公文生成任務需要多輪迭代
When  記錄高品味題庫
Then  提問欄位包含完整多輪對話
      且標註每輪的修改內容

E-005 時間戳欄位

Given 高品味題庫紀錄
When  填寫每個任務
Then  包含「起始時間」和「結束時間」欄位

F. 商業計價（4 項）

F-001 按任務定價 — 公文

Given 使用者完成一份公文生成（含多輪迭代）
When  session 結束
Then  計費 350 元臺幣/份

F-002 按任務定價 — 會議紀錄

Given 使用者完成一份會議紀錄生成
When  session 結束
Then  計費 200 元臺幣/份

F-003 用量統計 API

Given 廠商要查詢用量
When  GET /api/v1/usage?from=DATE&to=DATE
Then  回傳：
      { "total_tasks": number,
        "total_cost": number,
        "breakdown": [{ "skill": string, "count": number, "cost": number }] }

F-004 技能定價可配置

Given 系統管理者要調整定價
When  修改技能定價配置
Then  新定價立即生效
      且歷史帳單不受影響
